# 纯前端 JS 解析器架构

> 来源：PageHarvest 项目
> 日期：2026-07-20

## 架构决策：产品 vs 开源拆分

### 核心原则

| 层面 | 策略 | 理由 |
|------|------|------|
| **用户数据处理** | 纯浏览器端，不上传服务器 | 隐私合规，用户数据不经过第三方 |
| **解析器交付** | JS 代码，浏览器中运行 | 零服务器成本，用户打开页面即用 |
| **在线采集** | Python 公开代码，用户自部署 | 不背封 IP 的锅，用户自行承担风险 |
| **客户端工具** | exe（持续维护） | 本地可用，无需 Python 环境 |

### 设计思想

```
纯前端解析 = 产品          → 用户打开页面 → 浏览器解析 → 导出结果
在线采集 = 开源工具       → 用户自部署   → 服务器采集 → 本地处理
客户端 exe = 本地入口     → 下载即用     → 含浏览器解析能力
```

**不采用方案：** Issue Bot / GitHub Actions（用户数据经过第三方服务器，隐私问题）

## 1688 纯前端解析器

> 代码行数：约 1081 行（搜索页 + 详情页）

### 整体结构

```
parsers/1688/
├── search.js          — 搜索页解析（商品列表）
├── detail.js          — 详情页解析（单个商品）
├── shared.js          — 共享工具函数
├── config.js          — 平台参数配置
└── index.js           — 入口、路由
```

### 搜索页解析要点

- 列表项通过 CSS 选择器定位（非 API 数据）
- 商品卡片多层嵌套结构，需逐层提取
- 分页信息从 DOM 提取，支持翻页
- 图片 URL 补全（相对路径 → 绝对路径）

### 详情页解析要点

- 商品规格从 DOM 提取（非 JSON 内联数据）
- 属性名-值对从 `<table>` 或 `<dl>` 结构解析
- 价格区间处理（多规格对应多价格）
- 供应商信息提取

## 震坤行纯前端解析器

> 代码行数：约 599 行（搜索页 + 详情页）

### 搜索页

- 搜索结果通过 `window.__INITIAL_DATA__` JSON 嵌入 HTML（详见 [01-b2b-platforms-anti-crawl.md](01-b2b-platforms-anti-crawl.md#SSR-数据提取关键发现)）
- 前端解析器直接从该 JSON 提取，与后端 Playwright 方案不同
- 字段映射：`proSkuProductName`→标题、`proBrandName`→品牌、`proMaterialNo`→型号、`proSkuNo`→编码

### 详情页

- 商品详情字段在 DOM 中分散分布
- 规格参数从结构化 `<table>` 提取
- 价格信息在初始化数据中已有埋点

## 图片提取 4 级属性降级

当主要图片获取失败时，按以下顺序降级尝试：

| 等级 | 来源 | 说明 |
|------|------|------|
| 🥇 **primary** | 主图 URL | 从详情页 JSON/元素提取的主图 |
| 🥈 **secondary** | 备选图 URL | DOM 中其他可用图片元素 |
| 🥉 **thumbnail** | 缩略图 URL | 列表页的缩略图（质量较低） |
| 4️⃣ **placeholder** | 占位图 | 无图可用时输出空白/占位信息 |

每级尝试失败自动降级到下一级，确保解析不因图片缺失中断。

## 与后端采集的互补关系

| 维度 | 纯前端解析 | 后端 Playwright 采集 |
|------|-----------|-------------------|
| 数据路径 | 浏览器 → 用户 | 服务器 → 本地 |
| 隐私 | 数据不上传 | 需用户信任 |
| IP 限制 | 用户自己的 IP | 需代理 |
| 认证/滑块 | 用户自然通过 | 需绕行/等待 |
| 适用场景 | 快速查看、少量数据 | 批量采集、持续监控 |

共同点：解析器逻辑可复用（`lib/` → 双向参考）。
