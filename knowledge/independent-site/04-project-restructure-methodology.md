# 独立站项目文件重组方法

> 项目从草案→审批后，必须做一次文件结构清理，否则后面会越来越乱

## 触发条件

当项目出现以下信号时，就需要做一次文件重组：

- ✅ 关键审批通过（老板/客户点头）
- ✅ 技术架构确定（从「探索」切换到「执行」）
- ✅ 产生了大量草稿/测试文件
- ❌ 文档散落在不同目录，找东西靠搜索

## 重组三步走

### Step 1: 确立唯一入口

选一个文档做「主文档」，其他文档指向它或废弃。

**原则**：新加入项目的人，只需要读主文档就能知道全貌。

主文档应包含：
- 一句话项目描述
- 技术架构（已确定的）
- 目录结构图
- 阶段划分与当前阶段
- 各目录简要说明
- TODO

### Step 2: 归档废案

把以下文件移入 `archive/`：
- 技术选型阶段的淘汰方案（如「还是用 Next.js 吧」）
- 旧版计划（如「搭建步骤 v1」「时间线 v0」）
- 入口 README 改名或合并
- 工具清单（不再分立的）
- 任何不再更新且不会引用的文件

**注意**：归档不是删除，是为了「不干扰视野但能找回」。

### Step 3: 数据/资源/工具分离

重构后的目录结构示例：

```
项目根/
├── 项目主文档.md    ← 唯一入口
├── 产品/            ← 产品数据（CSV / 参数 / 图）
├── 设计/            ← 设计素材（品牌 / 色值 / 字体）
├── 竞品参考/        ← 竞品分析 / 截图
├── 工具/            ← 工具清单 / 脚本
├── 技术方案/        ← 基础设施方案 / 部署
├── 规划/            ← TODO / 阶段计划
├── 政策/            ← 法律 / 隐私政策
├── archive/         ← 废案（不删）
└── README.md        ← 简短指向主文档
```

## 为什么提前做

- 项目早期文件少，重构成本低
- 决策热乎，还能回忆「为什么选这个」
- 随着项目推进，新文件自然会落入正确的目录

## 容易被忽略的事

- **README 不是主文档** — README 指向主文档，主文档才是内容核心
- **时间线文档** — 建议用 TODO 或里程碑代替独立的时间线文档，否则容易过时
- **保留决策记录** — 归档时把「为什么淘汰」写在文件名或注释里，后续复盘有用

## 回顾

- Q: 做文件重组的触发条件是什么？
  A: 关键审批通过、技术架构确定、产生大量草稿文件、文档散落查找困难。
- Q: 重组三步走是哪三步？
  A: 确立唯一入口文档 → 归档废案（archive/） → 数据/资源/工具分离。
- Q: 主文档应包含哪些内容？
  A: 一句话描述、技术架构、目录结构图、阶段划分、各目录说明、TODO。
- Q: 归档和删除的区别是什么？
  A: 归档是移入 archive/ 不干扰视野但能找回，删除是彻底丢失。归档时在文件名或注释里记「为什么淘汰」。
- Q: README 和主文档的区别？
  A: README 是简短索引指向主文档，主文档才是内容核心。
