# 注：阶段化项目启动（Phase 0) 实战

## L1 术语

### `starred_at` [ⓘ]
GitHub Star API 返回的字段，记录你是什么时候 star 这个仓库的。
加上 `Accept: application/vnd.github.star+json` header 才会返回。
没有这个 header 也能获取仓库列表，但不知道哪些是新增的。

### `sha` 对比 [ⓘ]
Git 的哈希值，每次提交或文件变化都会生成。
增量同步时，比较上次记录的 sha 和当前的 sha，相同则无需更新。

### 原子写入 [ⓘ]
先写到临时文件 → 用 `os.replace()` 重命名替换原文件。
确保即使程序在写入中崩溃，原文件也不会损坏（要么全写，要么不写）。

## L2 概念

### 增量 vs 全量 [ⓘ]
| 模式 | 做法 | 适用 |
|------|------|------|
| 全量 | 每次都拉所有数据 | 首次，或数据量小 |
| 增量 | 只拉上次同步后的新增 | 日常维护，节省资源 |

增量同步的核心是**识别变化**——要么通过时间戳（`starred_at`），要么通过内容 Hash（`sha`）。

### 规则 vs AI 的分工 [ⓘ]
规则：快、确定、零成本——适合明确的分类（语言、话题重叠）。
AI：慢、不确定、有成本——适合语义理解（内容相关性、模糊匹配）。
正确策略：**规则做 80% 的事，AI 兜底剩下的 20%**。

## L3 架构

### 设计文档先行的价值 [ⓘ]
Phase 0 的核心不是写代码，是**想清楚再动手**。
15 个 Task 分解、模型设计、同步策略——都是在没有代码运行时就必须确定的决策。
写代码只是执行，真正的工程决策在设计阶段做。
