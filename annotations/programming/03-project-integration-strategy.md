# 注：子项目整合进父仓库的策略

## L1 术语

### `.guard.md` [ⓘ]
项目目录的访问权限标注文件，标记哪些目录可以自由修改、哪些需要审核。
本质是**沟通工具**：告诉团队（或 AI Agent）代码修改的边界在哪里。

### 自由开发目录 [ⓘ]
不需要审批即可修改的目录。对应"需要审核"的受控目录。
适合独立子项目、工具脚本等低风险模块。
与核心业务逻辑的受控目录形成分级管理。

## L2 概念

### 为什么两轮提交 [ⓘ]
子项目整合涉及**隐私安全**和**功能变更**两类完全不同的变更：
- 隐私安全：清理 Token、更新 .gitignore、删除 .venv
- 功能变更：移动代码、更新文档

混在一起会污染 git 历史，审计时难以追溯安全问题的修复。

## L3 架构

### 分层隔离的设计 [ⓘ]
```
PageHarvest/
├── .guard.md        ← 权限标注层
├── .gitignore       ← 安全边界层
├── factory-monitor/ ← 子项目隔离层
└── shared/          ← 公共模块层
```

每层解决一个问题：谁可以改、什么不能入库、各模块如何组织。
没有严格的架构规则，但约定的目录结构本身就是约束。
