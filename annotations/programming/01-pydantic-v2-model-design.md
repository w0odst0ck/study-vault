# 注：Pydantic v2 数据模型设计实战

## L1 术语

### `model_validator` [ⓘ]
Pydantic v2 的新验证机制，在字段级别验证完成后执行。
可以做跨字段校验——比如确保 `full_name == owner/name`。
比 `@validator`（每个字段独立验证）更适合需要字段间一致性的场景。

### `field_serializer` [ⓘ]
控制字段的序列化输出格式。比如：
- 时间对象 → ISO 字符串
- 自定义类 → 特定格式
- 敏感字段（如 token）→ 脱敏输出

### `min_length` / `max_length` [ⓘ]
Pydantic 的字段约束参数，定义字符串/列表的最小和最大长度。
属于**防御式编程**：在数据进入系统时就校验，尽早失败。

## L2 概念

### 衍生值不应持久化 [ⓘ]
设计中反复出现的模式：`average_confidence`、`orphans` 都是计算可得的值。
存了会产生数据一致性问题——源数据变了，衍生值没更新。
改为方法（`compute_orphans()`）每次调用时实时计算。

### 三层搜索路径 [ⓘ]
配置查找的优先级：
```
--config 参数 > ./star-vault.yaml > 内置默认值
```
最具体的（用户明确指定的）优先级最高，最泛用的（默认值）兜底。
这是 Unix 哲学在配置管理中的体现。

### 脱敏函数 [ⓘ]
```
"ghp_abcdef12345678" → "ghp_abcdef****5678"
```
不存 Token、不在日志中打印完整 Token、只显示首尾几位。
安全的最小原则：只有必要的人才看到必要的信息。
