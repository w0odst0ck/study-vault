# 注：1688 工厂监控全链路采集

## L1 术语

### alias ID [ⓘ]
1688 工厂的 memberId 有两种格式：
- 标准格式：`b2b-xxx` 开头
- 别名格式：英文单词（如 `wellfinesilicone`）
别名格式的工厂详情页无法通过标准 URL 模式访问，遍历时需要跳过。

### `data-btrack` 属性 [ⓘ]
1688 自定义 HTML 属性，用于前端埋点追踪。
因为 `<a>` 标签的 href 不可靠，改从 `data-*` 属性取数据。
`data-btrack="pc-card-shop-gallery-btn"` 是产品目录入口的标识。

## L2 概念

### 为什么 shopURL 不能直接当产品目录 URL [ⓘ]
shopURL 是工厂首页，产品目录页面是另一个 URL。
两者没有固定换算规则，必须从名片页 DOM 中提取。
这是 1688 的 SPA 路由设计——不同页面对应不同路由参数。

### UNIQUE 冲突 + 空字符串占位 [ⓘ]
SQLite UNIQUE 约束把空字符串 `''` 当作一个合法值。
多条记录的 shop_url 初始值都是 `''` → 第二条插入时 UNIQUE 冲突。
解决方案：初始值用 `<主键>`（每家不同），或允许 NULL。
这是数据库设计中最容易踩的坑之一。

## L3 架构

### 全链路流水线的设计思路 [ⓘ]
5 个阶段逐级递进，每阶段产出是下阶段的输入：

```
搜索（找工厂）→ 名片（找目录页）→ 产品目录（找商品）→ 过滤（筛灯类）→ 预警（对比变化）
```

每阶段可独立验证、独立重跑。即使第 3 阶段挂了，第 1-2 阶段的数据不会丢失。

### 验证驱动开发 [ⓘ]
先写 verify.py 逐步骤验证 DOM 选择器 → 验证通过提取为正式脚本 → 保留 verify.py 作为选择器参考。
网页 DOM 结构经常变，验证脚本相当于**活的 API 文档**。
