# 数据治理 / 数据中台 模拟面试题（第二套）

## 一、MySQL 数据库设计与治理（5 题）

### 1. 在数据中台场景下，MySQL 分库分表的常用方案有哪些？水平分表后如何解决全局唯一 ID、跨库分页、事务问题？

### **参考答案**

1. **分库分表方案**

- 水平拆分：按用户 ID 哈希、按日期范围、按区域拆分。
- 垂直拆分：按业务域拆分（用户库、订单库、商品库）。
- 中间件：Sharding-JDBC、MyCat。

2. **全局唯一 ID**

- 雪花算法、百度 UidGenerator、Leaf。

3. **跨库分页**

- 全局统一排序字段（如 create_time），避免深度分页。
- 业务上限制分页页数，或使用游标 / 位置标记分页。
- 二次聚合：各分片返回局部结果，应用层合并排序。

4. **分布式事务**

- 最终一致性：Seata AT/TCC、可靠消息最终一致性。
- 不追求强一致，中台场景多用最终一致。

### 2. 什么是索引失效？请列举至少 6 种常见场景，并结合数据中台 SQL 开发说明如何避免。

### **参考答案**

1. **索引失效场景**

- 对索引字段使用函数、运算（`substring(name)`、`age+1`）。
- 隐式类型转换（字符串数字与数字比较）。
- `like '%xxx'` 左模糊。
- 使用 `NOT IN`、`!=`、`IS NOT NULL` 导致范围查询失效。
- 联合索引不满足最左前缀原则。
- `OR` 连接的条件一侧无索引。
- 数据分布不均，优化器直接走全表。

2. **避免方式**

- SQL 上线前执行`EXPLAIN`审核。
- 禁止在索引列上运算或函数处理。
- 规范联合索引顺序，高频查询在前。
- 统一字段类型，避免隐式转换。
- 禁止无 where 条件的更新删除。

### 3. MySQL 主从复制原理是什么？在数据同步到数仓时，如何利用主从架构降低对业务库影响？

### **参考答案**

1. **主从原理**

- 主库 binlog 记录变更 → 从库 IO 线程拉取 → 中继日志 → SQL 线程重放。

2. **降低业务库影响**

- 数仓抽取直接从**从库**读取，不查主库。
- 全量同步放在从库低峰期执行。
- 使用 CDC（Canal/Flink CDC）监听 binlog，非侵入采集。
- 控制抽取并发与批次，避免从库压力过高。

### 4. 数据中台需要对 MySQL 业务库做数据标准治理，具体包括哪些内容？如何落地？

### **参考答案**

1. **标准治理内容**

- 命名规范：库、表、字段、索引统一命名。
- 数据类型规范：长度、精度、枚举统一。
- 字典标准：性别、状态、渠道等枚举值统一。
- 主键、唯一键、外键（逻辑外键）规范。
- 通用字段：`id`、`create_time`、`update_time`、`is_deleted`。

2. **落地方式**

- 建表必须走数据平台审批。
- 自动扫描违规表结构并告警。
- 历史表逐步改造，建立数据字典。
- 元数据管理平台统一维护。

### 5. 什么是脏读、不可重复读、幻读？MySQL 的事务隔离级别如何选择？中台业务如何平衡一致性与性能？

### **参考答案**

1. **问题定义**

- 脏读：读到未提交数据。
- 不可重复读：同一事务内两次查询结果不一致。
- 幻读：范围查询出现新增行。

2. **隔离级别**

- Read Uncommitted（极少用）
- Read Committed（RC，互联网常用）
- Repeatable Read（RR，默认）
- Serializable（强一致，性能差）

3. **中台选择**

- 业务库写操作多用 RC，平衡并发与一致性。
- 数仓同步不依赖事务，依赖 binlog 位点保证最终一致。
- 关键核心订单等场景可使用 RR 或分布式事务。

## 二、Hive 数仓建模与分布式数据处理（5 题）

### 1. 请对比维度建模中的**星型模型**与**雪花模型**，并说明在数据中台建设中为什么更常用宽表模型？

### **参考答案**

1. **星型模型**

- 事实表 + 多个冗余维度表，无多层关联，查询快。

2. **雪花模型**

- 维度表继续规范化，多层关联，减少冗余，查询慢。

3. **宽表更适合中台原因**

- Hive 分布式计算适合大表扫描，不适合大量 Join。
- 宽表减少 Shuffle，提升 ETL 与查询效率。
- 统一口径，减少多表关联歧义。
- 便于数据服务、自助分析、API 输出。

---

### 2. Hive 动态分区与静态分区区别？在数仓增量同步中如何使用动态分区？有哪些坑？

### **参考答案**

1. **区别**

- 静态分区：手动指定分区值。
- 动态分区：根据字段值自动创建分区。

2. **使用场景**

- 按天 / 小时接入 ODS 层。
- 增量数据按日期自动分区。

3. **常见问题**

- 分区爆炸：小文件过多。
- 数据倾斜：某分区数据过大。
- 性能消耗：动态分区会触发额外 Shuffle。

4. **优化**

- 限制分区数量，合并小文件。
- 开启`hive.merge.mapfiles`。
- 先按分区聚合再写入。

---

### 3. 数仓开发中常见的缓慢变化维（SCD）有哪些类型？在中台用户 / 商品维度表中如何落地？

### **参考答案**

1. **SCD 类型**

- SCD1：直接覆盖更新。
- SCD2：新增版本记录，start_date/end_date/is_current。
- SCD3：增加历史字段，保留少量历史。

2. **中台落地**

- 简单属性用 SCD1。
- 重要变化（用户等级、商品状态）用 SCD2。
- 通过`row_number() partition by ... order by`实现拉链表。
- 每日生成快照，便于历史回溯分析。

---

### 4. 请详细说明数仓开发中小文件问题的危害、产生原因及完整解决方案。

### **参考答案**

1. **危害**

- NameNode 压力大。
- 查询启动慢、并行度过高、执行低效。
- 严重影响 Hive 稳定性。

2. **原因**

- 高频增量写入、动态分区、多并行 Reduce。

3. **解决方案**

- 写入前聚合，减少文件数。
- 使用`Distribute By`分区。
- 配置参数合并小文件。
- 定时任务后台合并。
- 使用 ORC/Parquet 列式存储。
- Spark/Flink 写入控制文件大小。

### 5. 数据中台指标体系建设中，如何保证指标口径一致？如何通过数仓分层实现指标可复用？

### **参考答案**

1. **口径一致**

- 统一维度：用户、商品、时间、区域。
- 统一事实表：避免多源头计算。
- 指标字典管理，统一命名与公式。
- 血缘分析，防止指标重复定义。

2. **分层复用**

- DWD：明细统一清洗。
- DWS：按主题公共聚合。
- ADS：面向应用定制指标。
- 公共指标下沉到 DWS，避免重复计算。

## 三、向量数据库 Milvus（4 题）

### 1. 向量检索中的精确搜索与近似搜索（ANN）区别是什么？Milvus 为什么默认使用 ANN？

### **参考答案**

1. **精确搜索**

- 暴力对比所有向量，100% 准确，但速度慢，不适合大数据量。

2. **近似搜索（ANN）**

- 通过聚类、图索引等加速，牺牲少量精度换取巨大速度提升。

3. **Milvus 默认 ANN 原因**

- 面向高维、海量向量场景。
- 满足推荐、检索、RAG 等业务低延迟要求。
- 工程上可平衡召回率与性能。

---

### 2. Milvus 中 HNSW、IVF_FLAT、IVF_SQ8、SCANN 索引分别适用什么中台场景？

### **参考答案**

- **FLAT**：小数据量、要求 100% 精确。
- **IVF_FLAT**：中等规模，平衡精度与速度。
- **IVF_SQ8**：向量压缩，节省内存，高并发。
- **HNSW**：低延迟查询，用户画像、实时检索。
- **SCANN**：极高吞吐、大规模向量，推荐系统。

---

### 3. 如何实现数仓（Hive/Spark）批量向量生成并写入 Milvus？如何保证增量更新与幂等性？

### **参考答案**

1. **流程**

- 数仓 DWD 层取出文本 / 图像 / 用户特征。
- Spark 调用 Embedding 模型生成向量。
- 通过 Milvus SDK 批量写入。
- 建立业务主键与向量 ID 映射表。

2. **幂等性**

- 按主键去重，重复写入覆盖。
- 按日期分区增量同步。
- 记录同步水位，避免重复同步。

---

### 4. 在 AI 数据中台里，Milvus 如何与大模型 RAG 结合？从文档入库到问答的完整流程是什么？

### **参考答案**

1. **流程**

- 文档接入 ODS → 清洗切块 → DWD 文本块。
- Embedding 生成向量 → 写入 Milvus。
- 用户提问 → 向量化 → Milvus 召回相似文本。
- 上下文 + 问题送入大模型 → 生成回答。

2. **中台价值**

- 统一向量管理。
- 统一文档治理与权限。
- 检索效果可监控可迭代。

## 四、综合架构题（1 题）

**参考答案**

1. **数据流向**
    
    MySQL 业务库 → CDC / 抽取 → ODS → DWD（清洗）→ DWS（聚合）→ ADS（指标）+ Milvus（向量）。

    DolphinScheduler/Airflow 定时调度，依赖链管理。

    空值、重复、值域、行数监控、指标对账、延迟监控。
    
4. **治理体系**
    
    元数据管理、数据标准、数据字典、数据权限、数据血缘、生命周期管理。
    
5. **服务层**
    

- 指标服务 API
- 向量检索服务
- 自助分析平台
- 报表与大屏
