# 一、MySQL 与业务数据治理（5 题）

## 1

在数据中台同步 MySQL 业务库时，业务表**没有 update_time、没有自增 ID、没有 binlog**，只有业务主键。

你如何设计增量同步方案，保证不丢数据、不重复同步？

### **参考答案**

1. 采用**全量比对 + 业务主键哈希**方案：
    
    - 首次全量抽取，以业务主键构建唯一键映射。
    
2. 后续每次抽取全量主键列表，与上一次快照**取差集**得到新增 / 更新数据。
3. 对关键字段做 MD5 哈希，用于判断记录是否变更。
4. 数仓层按主键去重，使用 `row_number()` 保证幂等。
5. 治理层面推动业务整改：强制增加 `create_time`/`update_time`，规范主键设计。

## 2

联合索引 `(a,b,c)`，请写出哪些 SQL 能命中索引，哪些会失效，并说明原因。

1. `where a=? and b>? and c=?`
2. `where b=? and a=?`
3. `where a=? order by b,c`
4. `where a like 'x%' and b=?`
5. `where a=? or c=?`

### **参考答案**

1. 部分命中：`a,b` 有效，`c` 失效（范围查询中断后缀）
2. 命中：优化器会调整顺序，仍满足最左前缀
3. 命中：覆盖排序，无需 filesort
4. 命中：前缀模糊不影响左侧索引
5. 失效：`OR` 导致索引无法整体生效

## 3

MySQL 大表（千万～亿级）要接入数据中台，直接 `select *` 会卡死主库。

请给出**安全、低侵入、可断点续传**的全量抽取方案。

### **参考答案**

1. **从从库抽取**，避免影响主库。
2. **按主键分段分页**，禁用 `limit offset`，使用 `id > ?` 滚动拉取。
3. 控制批次大小（如 5000～10000），降低单次 IO 压力。
4. 记录每次抽取最大 ID，实现断点续传。
5. 低峰期执行，配合 `innodb_read_io_threads` 适度调优。
6. 最终改用 CDC 方案长期替代全量抽取。

## 5

业务频繁 `update` 大字段（文本 / JSON），导致同步到数仓时出现**大量重复计算、链路延迟、存储膨胀**。

请给出治理方案。

### **参考答案**

1. 识别**更新频繁但分析价值低**的字段，数仓层不同步或单独存储。
2. 对大字段做**哈希标记**，仅哈希变化时才同步。
3. 使用 CDC 只订阅真实变更字段，而非整行覆盖。
4. 数仓层做增量合并，避免每日全量覆盖。
5. 推动业务垂直拆分冷热字段，减少不必要更新。

# 二、Hive 数仓建模 & 分布式计算（5 题）

## 1

请完整设计一套**用户行为主题数仓模型**，从 ODS → DWD → DWS → ADS，每层用途、主键、分区、存储格式、典型宽度。

### **参考答案**

- ODS：原始日志，按 `dt` 分区，TEXT/ORC，结构同业务
- DWD：清洗明细，主键 `log_id`，去重、脱敏、规范枚举
- DWM：中间层，行为漏斗、会话聚合
- DWS：用户日粒度宽表，`user_id,dt`，聚合各类行为指标
- ADS：面向业务报表 / 大屏，如 DAU、留存、转化率
    
    存储统一 ORC/Parquet，压缩 ZLIB，按天分区

## 2

写出一段**完整的用户 7 日留存 SQL**，要求规范、可上线、支持重复执行。

### **参考答案**

```sql
WITH user_first AS (
    SELECT user_id, MIN(dt) AS first_dt
    FROM dwd_user_behavior_d
    WHERE dt BETWEEN '2025-01-01' AND '2025-01-07'
    GROUP BY user_id
)
INSERT OVERWRITE TABLE ads_user_retention PARTITION (dt='2025-01-07')
SELECT
    first_dt,
    COUNT(DISTINCT uf.user_id) AS new_user_cnt,
    COUNT(DISTINCT ub.user_id) AS retain_7d_cnt,
    ROUND(COUNT(DISTINCT ub.user_id)/COUNT(DISTINCT uf.user_id),4) AS retention_7d
FROM user_first uf
LEFT JOIN dwd_user_behavior_d ub
    ON uf.user_id = ub.user_id
    AND ub.dt = DATE_ADD(uf.first_dt,7)
GROUP BY first_dt;
```

## 3

ETL 任务执行 2 小时跑不完，定位是**数据倾斜**。

请写出你完整的排查思路 + 对应优化手段（至少 4 种）。

### **参考答案**

排查思路：

1. 看 Yarn 监控：个别 reducer 耗时远高于其他
2. 打印 key 分布，找出热点 key
3. 判断是 group by 倾斜还是 join 倾斜
4. 查看是否为空值、特殊值、超级用户导致

优化手段：

1. 过滤异常 key
2. 热点 key 加盐打散
3. 开启 `hive.groupby.skewindata`
4. 大表 join 小表转为 map join
5. 拆分 SQL：先聚合再 join
6. 调整并行度与内存配置

## 4

数仓开发中**拉链表、快照表、全量同步、增量同步**分别适用什么场景？如何选择？

### **参考答案**

- 拉链表（SCD2）：维度需历史回溯，如用户等级、商品状态
- 快照表：每日全量镜像，适合小维度、无需历史明细
- 全量同步：小表、无增量字段、简单业务
- 增量同步：大表、高频更新、追求性能与时效

选择依据：数据量、更新频率、是否需要历史版本、存储成本、计算成本。

# 三、向量数据库 Milvus 与 AI 中台（4 题）

## 1

在企业数据中台中，**向量数据库应该放在数据链路的哪一层**？为什么？

ODS / DWD / DWS / ADS / 服务层

### **参考答案**

放在 **DWS 及服务层**。

- DWD 提供清洗后的主体与文本特征
- DWS 做向量化、Embedding 生成、向量聚合
- Milvus 作为服务层向量索引，对外提供检索
    
    不适合放在下层：向量计算昂贵、存储大、下层只需结构化治理。

---

## 2

Milvus 出现**召回率低、检索不准、QPS 上不去、内存爆了**，分别怎么优化？

### **参考答案**

- 召回率低：提高 nprobe、调整 nlist、改用 HNSW 降低 ef_construction
- 检索不准：Embedding 模型质量差、文本切块不合理、相似度度量方式不匹配
- QPS 上不去：集群扩容、分片、调整 Segment 大小、使用 GPU
- 内存爆了：改用 IVF_SQ8 量化、降低向量维度、冷热分离、部分集合不常驻内存

---

## 3

如何实现**数仓与 Milvus 的数据一致性**？

比如用户画像在 Hive 中更新了，但向量库没更，导致检索错误。

### **参考答案**

1. 统一主键：`user_id` 双向映射
2. 建立**向量更新水位表**，记录同步版本 / 时间
3. 增量任务：每日从 DWS 抽取变化用户，更新向量
4. 幂等写入：重复插入自动覆盖
5. 每日对账：Hive 用户数 ≈ Milvus 主键数量
6. 全量重建 + 双写切换机制，保证最终一致

## 4

你正在搭建**RAG 知识库中台**，请设计从「文档上传 → 数仓治理 → 向量检索 → 问答服务」的完整架构。

### **参考答案**

1. 文档上传 → 解析（PDF/Word）→ 文本切块 → ODS
2. DWD 清洗：去重、去噪、分段、标准化
3. DWS 生成 Embedding，存入 Milvus

5. 治理：元数据管理、权限、版本、检索效果监控
6. 质量指标：召回率、准确率、响应时间

# 四、综合架构 & 治理题（1 题）

## 请设计一套**企业级数据中台数据质量平台**核心功能，要求覆盖：

完整性、准确性、一致性、唯一性、及时性、有效性，并说明如何落地。

### **参考答案**

1. **规则引擎**
    
    - 非空、值域、枚举、格式、行数波动、主键重复
    
2. **监控体系**
    
    - 表级行数、字段空值率、指标环比、产出延迟
    
3. **对账能力**
    
    - 业务库与数仓对账、跨层指标对账、数仓与向量库对账
    
4. **告警与处置**
    
    - 分级告警、自动屏蔽、自动重跑、人工派单
    
5. **报表与大盘**
    
    - 质量评分、治理覆盖率、问题趋势
    
6. **落地路径**
    - 先核心表 → 核心指标 → 全量覆盖
    - 与调度系统联动，质量不达标不下发
