# 数据治理 / 数据中台岗位 模拟面试题（数据库 + 数仓 + 向量数据库）

## 一、MySQL 数据库设计与数据治理（4 题）

### 1. 请简述在数据中台场景下，MySQL 业务库表设计的核心原则，以及如何避免过度冗余与查询性能问题？

### **参考答案**

1. **核心设计原则**
    
    - 遵循三范式减少冗余，同时根据中台查询场景适度反范式；
    - 主键统一使用自增 ID 或雪花 ID，避免业务字段做主键；
    - 合理设计索引：高频查询字段建索引，避免冗余索引、重复索引；
    - 字段类型精简：用 tinyint/int 替代字符串，时间用 datetime/timestamp；
    - 增加通用审计字段：`id、create_time、update_time、creator、is_deleted`。
    
2. **冗余与性能平衡**
    
    - 关联查询频繁的小表适度冗余字段，减少多表 join；
    - 大表拆分：垂直拆分（冷热字段分离）、水平拆分（按时间 / 用户 ID 分表）；
    - 使用中间表 / 汇总表提前聚合数据，支撑中台报表与服务接口。
    

---

### 2. 什么是 MySQL 慢查询？在数据治理中如何定位、优化慢 SQL 并建立治理规范？

### **参考答案**

1. **慢查询定义**
    
    执行时间超过阈值（如 1s）、扫描行数过多、未命中索引、锁等待严重的 SQL。
    
2. **定位方式**
    
    - 开启慢查询日志，结合 `explain`/`show profile` 分析执行计划；
    - 查看 key、type、rows、Extra 字段判断索引使用情况；
    - 使用监控工具（Prometheus+Grafana、SkyWalking）实时监控。
    
3. **常见优化**
    
    - 建立合适索引，避免 `select *`、避免 `like %xx%`、避免隐式类型转换；
    - 减少大事务、拆分复杂子查询为 join 或临时表；
    - 大分页使用 `延迟关联`/`游标分页`。
    
4. **治理规范**
    
    - SQL 上线必须审核；
    - 禁止无索引条件更新 / 删除；
    - 定期清理无效索引、冗余数据；
    - 建立慢查询周通报与整改机制。
    

---

### 3. 数据中台常需要从 MySQL 同步数据到数仓，如何保证数据一致性、增量同步与幂等性？

### **参考答案**

1. **增量同步方案**
    
    - 基于 `update_time` 增量抽取；
    - 基于 binlog 实时采集（Canal、Flink CDC）。
    
2. **一致性保障**
    
    - 全量同步加锁或使用快照保证一致性；
    - 增量同步记录位点（offset），异常重启可续跑。
    
3. **幂等性设计**
    
    - 以业务主键 / 联合唯一键作为数仓主键；
    - 同步任务支持重复执行覆盖写入；
    - 增加数据版本号或时间戳，避免旧数据覆盖新数据。
    

---

### 4. MySQL 在数据治理中的常见数据质量问题有哪些？如何建立监控与清洗规则？

### **参考答案**

1. **常见问题**
    
    - 空值、异常值（如负数金额、超大时间）；
    - 数据重复、主键冲突；
    - 字段含义不一致、枚举值混乱；
    - 跨表关联字段类型 / 长度不匹配。
    
2. **治理手段**
    
    - 建表时增加非空约束、唯一约束、检查约束；
    - 定时任务校验数据规则，生成质量报告；
    - 建立数据标准：字段命名、枚举字典、单位统一；
    - 使用数据质量平台（如 Great Expectations）自动化监控告警。

## 二、Hive 数仓建模与分布式数据处理（4 题）

### 1. 请介绍数据中台标准数仓分层模型（ODS-DWD-DWS-ADS），每层职责与设计要点？

### **参考答案**

1. **ODS 层**
    
    - 职责：原始数据接入，与业务库结构基本一致；
    - 要点：保留原始数据，不做清洗，按天 / 小时分区。
    
2. **DWD 层**
    
    - 职责：明细层，数据清洗、脱敏、规范化；
    - 要点：统一字段标准，去除空值 / 脏数据，构建宽明细。
    
3. **DWS 层**
    
    - 职责：汇总层，按主题轻度聚合（用户、商品、订单）；
    - 要点：按维度聚合，减少重复计算，支撑上层应用。
    
4. **ADS 层**
    
    - 职责：应用层，面向报表、大屏、接口直接输出；
    - 要点：定制化聚合，性能优先，结构简单。
    

---

### 2. 数仓建模常用方法：维度建模（星型 / 雪花）与范式建模区别？中台更推荐哪种，为什么？

### **参考答案**

1. **区别**
    
    - **范式建模**：减少冗余，适合事务型库，查询需大量 join，复杂且慢；
    - **维度建模**：以事实表为中心，关联维度表，结构简单、查询高效。
    
2. **中台推荐维度建模**
    
    - 数仓以分析查询为主，追求读性能；
    - 星型模型更适合 Hive 等分布式引擎，减少 shuffle；
    - 业务理解成本低，便于数据服务、自助分析。
    

---

### 3. Hive 数据倾斜是什么？常见场景及在数据中台 ETL 中的解决办法？

### **参考答案**

1. **数据倾斜**
    
    个别 key 数据量远大于其他 key，导致部分 reducer 卡死，任务极慢。
    
2. **常见场景**
    
    - group by 某 key 分布不均（如空值、特殊值）；
    - join 时大表 key 集中；
    - count distinct 热点值。
    
3. **解决方案**
    
    - 预处理：过滤空值 / 异常 key；
    - 加盐打散：`concat(key, rand())` 拆分热点；
    - 参数调优：`hive.groupby.skewindata=true`；
    - 大表 join 小表使用 map join；
    - 拆分 SQL：先聚合再关联。
    

---

### 4. 数据中台如何保证数仓数据质量？包括时效性、准确性、完整性、一致性。

### **参考答案**

1. **时效性**
    
    - 任务调度依赖链管理（Airflow/DolphinScheduler）；
    - 任务超时告警、延迟监控。
    
2. **准确性**
    
    - 字段规则校验：值域、格式、枚举范围；
    - 关键指标对账：业务库与数仓指标对比。
    
3. **完整性**
    
    - 记录行数监控，环比 / 同比异常告警；
    - 必选字段非空校验。
    
4. **一致性**
    
    - 统一维度口径（用户、商品、时间）；
    - 跨主题指标定义一致；
    - 血缘管理，避免指标重复定义。

## 三、向量数据库 Milvus 与 AI 数据中台结合（3 题）

### 1. 什么是向量数据库？Milvus 在数据中台的应用场景有哪些？

### **参考答案**

1. **向量数据库**
    
    专门存储、检索高维向量（Embedding），支持高效近似最近邻搜索（ANN），用于相似度匹配。
    
2. **中台应用场景**
    
    - 文本 / 图像相似检索（智能问答、内容推荐）；
    - 用户向量画像、个性化推荐；
    - 非结构化数据治理（文档聚类、去重）；
    - 结合大模型构建 RAG 检索增强生成。
    

---

### 2. Milvus 核心组件与架构是什么？如何选择索引类型以适配中台高并发查询？

### **参考答案**

1. **核心架构**
    
    - Root Coord：集群管理；
    - Query Coord/Query Node：查询执行；
    - Index Coord/Index Node：构建索引；
    - etcd：元数据存储；
    - 对象存储：向量数据持久化。
    
2. **常用索引及选择**
    
    - **FLAT**：精准搜索，数据量小使用；
    - **IVF_FLAT**：平衡速度与精度，通用场景；
    - **IVF_SQ8**：压缩存储，节省内存，中台高并发推荐；
    - **HNSW**：查询速度极快，适合低延迟服务，但内存占用高。
    

---

### 3. 如何将 Milvus 接入数据中台？从数仓到向量库的完整流程是什么？

### **参考答案**

1. **流程**
    
    - DWD/ODS 层抽取非结构化数据（文本、图像）；
    - 通过大模型 / Embedding 模型生成向量；
    - 使用 Spark/Flink 批量写入 Milvus；
    - 建立向量与业务 ID 关联映射，保证可回溯；
    - 对外提供统一向量检索 API。
    
2. **治理要点**
    
    - 向量版本管理；
    - 向量库与数仓数据一致性校验；
    - 向量召回质量监控（准确率、召回率）；
    - 定时增量更新与全量重建机制。

## 四、综合题（1 题）

### 请设计一套从 MySQL 业务库 → Hive 数仓 → Milvus 向量库的数据中台流转方案，并说明数据治理关键点。

### **参考答案**

1. **数据流转链路**
    
    - MySQL 业务库 → Canal/Flink CDC → ODS 层；
    - ODS 清洗 → DWD 明细宽表；
    - DWD 生成文本 / 用户特征 → Embedding 向量化；
    - 向量 + 业务主键 → Milvus；
    - DWS/ADS 支撑报表与指标服务，Milvus 支撑检索服务。
    
2. **治理关键点**
    
    - 统一数据标准、字段字典；
    - 全链路数据血缘与血缘监控；
    - 数据质量规则全覆盖；
    - 增量同步幂等性；
    - 向量库与数仓数据一致性；
    - 权限、脱敏、分级分类管理。
