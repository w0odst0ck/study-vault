## 第 1 题

**问题**：请结合你在工厂数据中台项目，详细说明你是如何设计**ODS / DWD / DWS / ADS**四层数仓架构的？每一层的职责、数据来源、粒度、更新策略分别是什么？

### **参考答案**：

1. **ODS 层**
    
    - 职责：原样接入业务源数据，不做业务加工。
    - 来源：ERP、MES、WMS、Excel 等多源异构数据。
    - 粒度：与业务系统一致，单条明细。
    - 更新：全量初始化 + 增量同步（按日期 / 时间戳），保留历史分区。
    
2. **DWD 层**
    
    - 职责：清洗、脱敏、去重、统一编码、关联维度，形成标准业务明细。
    - 粒度：最细业务过程粒度，如单条生产记录、单条出入库、单条订单。
    - 处理：空值填充、异常值剔除、维度退化、统一字段名。
    - 更新：按天 / 按小时增量刷新。
    
3. **DWS 层**
    
    - 职责：轻度聚合，按主题构建公共汇总宽表。
    - 主题：生产、库存、销售、设备、订单等。
    - 粒度：按天 + 产品 + 车间 + 产线等多维度聚合。
    - 用途：为 ADS 和报表提供统一口径，避免重复计算。
    
4. **ADS 层**
    
    - 职责：面向应用的指标层，输出最终报表 / 大屏 / 接口指标。
    - 示例：日产量、合格率、库存周转率、订单履约率。
    - 更新：T+1 为主，核心看板支持准实时。
    

整体设计目标：**口径统一、复用性强、性能高、易维护**。

## 第 2 题

**问题**：你在项目中使用了**DataWorks 进行 ETL 开发与调度**，请描述一个典型的任务流设计：从同步、清洗、加工到产出报表指标的完整链路，并说明你如何做**任务依赖、失败重试、告警**。

### **参考答案**：

1. **典型任务链路**
    
    - 数据同步节点：MySQL → ODS 层。
    - 数据质量校验节点：空值、重复值、范围校验。
    - Hive SQL 节点：ODS → DWD 清洗加工。
    - 聚合节点：DWD → DWS 宽表构建。
    - 指标计算节点：DWS → ADS 指标表。
    - 导出 / 接口节点：供可视化平台查询。
    
2. **任务依赖**
    
    - 按数据流向设置**上游依赖**，保证数据先有下层再有上层。
    - 同层任务并行，跨层串行，提升执行效率。
    
3. **失败与告警**
    
    - 失败自动重试 2～3 次，避免临时抖动。
    - 仍失败则终止下游，避免脏数据扩散。
    - 配置钉钉 / 邮件告警，包含任务名、错误信息、运行时间。
    - 关键任务设置超时监控，防止卡死占用资源。

## 第 3 题

**问题**：在实际数仓开发中，**数据倾斜**是高频问题。请说明你遇到过哪些倾斜场景（如 Join、Group By），并给出具体的解决方法和在项目中的实际效果。

### **参考答案**：

1. **Group By 倾斜**
    
    - 场景：按产品、车间统计时，少数爆款产品 / 核心车间数据量极大。
    - 解决：
        
        - 开启 Map 端预聚合。
        - 对热点 key 加随机盐，分两阶段聚合。
        
    - 效果：单个 Reduce 任务从几十分钟降到几分钟。
    
2. **Join 倾斜**
    
    - 场景：大表 Join 时，某关联键值过多（如大量相同物料编码）。
    - 解决：
        
        - 小表使用 MapJoin 广播。
        - 热点 key 单独拆分 Join，非热点正常 Join 后合并。
        - 空值 / 异常 key 随机化打散。
        
    
3. **项目实际效果**
    
    - ETL 整体运行时间缩短 40%～60%。
    - 解决了任务经常卡住、超时失败问题。
    - 集群资源利用率更均衡，减少资源浪费。

## 第 4 题

**问题**：你简历中提到了**Milvus 向量数据库 + RAG 检索增强**，请解释：

1）为什么要用向量检索；2）BM25 与向量检索的区别；3）你在项目中如何做召回、排序、优化准确率。

### **参考答案**：

1. **为什么用向量检索**
    
    - 传统关键词匹配（BM25）只能做字面匹配，无法理解语义。
    - 向量检索可以理解句子含义，适合法律、法规、文档类相似性检索。
    
2. **BM25 vs 向量检索**
    
    - BM25：关键词权重匹配，速度快，适合精确术语、编号、专有名词。
    - 向量检索：语义相似度，适合意图理解、相似问题、模糊查询。
    
3. **项目中 RAG 优化流程**
    
    - 文档切块：按语义分段，保留上下文，设置少量重叠。
    - 向量化：使用中文 embedding 模型生成向量存入 Milvus。
    - 召回：BM25 关键词召回 + Milvus 向量召回多路召回。
    - 排序：对召回结果做交叉重排，取 Top-K 最相关片段。
    - 优化：过滤低相似度结果、按时间 / 有效性过滤、去重。
    - 结果：问答准确率明显提升，回答更贴合专业内容。

## 第 5 题

**问题**：如果你要从零开始搭建一套**企业级数据治理体系**，你会优先做哪几件事？请按优先级排序，并说明每一步的产出物。

### **参考答案**：

1. **梳理核心业务指标与口径（优先级最高）**
    
    - 产出：指标字典、口径文档、指标 Owner。
    - 目的：先统一 “大家说的是不是同一个数”。
    
2. **建立数据分层规范与命名规范**
    
    - 产出：数仓规范文档、表名字段规范、分层标准。
    - 目的：避免混乱，保证后续可扩展。
    
3. **搭建数据质量监控体系**
    
    - 产出：完整性、准确性、一致性、及时性规则。
    - 目的：提前发现问题，而不是业务投诉后补救。
    
4. **元数据与数据血缘建设**
    
    - 产出：数据字典、血缘图谱、表级 / 字段级血缘。
    - 目的：快速定位问题、评估变更影响、实现可追溯。
    
5. **数据权限与安全治理**
    
    - 产出：数据分级分类、权限申请流程、脱敏规则。
    - 目的：满足合规，防止数据泄露。
    

整体思路：**先统一口径，再规范架构，再建质量，最后做权限与合规**，用最小成本最快看到治理价值。
