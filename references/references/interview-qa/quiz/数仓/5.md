# 一、MySQL 与业务数据治理（5 题）

## 1

业务 MySQL 一张 2 亿订单大表，无分区、无分表、无规范更新字段，现在要接入中台实时数仓，业务不允许改表、不允许停服。

请给出**可上线、可运维、可治理**的完整接入方案。

### **参考答案**

1. **接入方式**
    
    - 基于 binlog 接入 Canal/Flink CDC，不侵入业务
    - 全量历史数据从从库分批拉取，按主键 id 滚动同步
    
2. **增量与一致性**
    
    - 全量 + 增量合并，通过 CDC 位点保证不丢不重
    - 数仓层按订单 id 去重，保证幂等
    
3. **治理与后续**
    
    - 建立元数据标签，标记为核心高风险表
    - 推动后续按日期分区、清理冷数据
    - 禁止全表扫描，SQL 上线强制审核

## 2

联合索引失效你已经知道，现在问：

**MySQL 优化器在什么情况下会主动放弃索引而选择全表扫？**

中台同步任务因此导致慢 SQL，如何解决？

### **参考答案**

优化器选择全表扫描常见场景：

1. 查询结果占比过高（如超过 20%~30%）
2. 索引区分度极低（性别、状态等）
3. 统计信息过时，innodb 索引基数不准
4. limit 过大或深度分页
5. 使用函数、隐式转换导致索引无法有效使用

解决：

- 强制索引 `FORCE INDEX`
- analyze table 更新统计信息
- 改造 SQL，缩小查询范围
- 建立更合适的组合索引

## 3

中台同步任务频繁出现**主从延迟导致的数据不一致**，比如刚插入就同步，读从库读不到。

你如何从**架构、同步逻辑、治理规范**三层解决？

### **参考答案**

1. **架构**
    
    - 实时场景走 CDC binlog，不走 SQL 查询
    - 关键任务读主库或半同步从库
    
2. **同步逻辑**
    
    - 增加时间偏移，延迟几秒再抽取
    - 按位点 / GTID 同步，不依赖时间
    
3. **治理规范**
    
    - 禁止依赖瞬间一致性，中台统一最终一致
    - 建立对账机制，异常自动修复

## 4

什么是**数据闭环治理**？MySQL 业务库如何实现 “建表规范→变更审核→质量监控→整改闭环”？

### **参考答案**

1. **建表规范**
    
    强制主键、更新时间、通用字段、注释、索引规范
2. **变更审核**
    
    平台化建表，DDL 自动校验，违规拦截
3. **质量监控**
    
    自动巡检：无注释、无索引、大表、膨胀表
4. **整改闭环**
    
    工单派单→限期整改→验收通过→关闭
    
    与部门指标、owner 考核挂钩

## 5

业务频繁使用 `text/json` 大字段，导致：

同步慢、存储高、数仓清洗难、索引无效。

请给出**技术治理方案 + 业务推动方案**。

### **参考答案**

技术：

- 大字段剥离到单独表，减少主表 IO
- JSON 提取常用字段冗余，建立索引
- 同步时只抽取必要字段，不抽全量
    
    业务推动：
- 制定大字段使用规范
- 超过一定长度必须走对象存储
- 数仓不负责存储原始大文本，只存索引信息

# 二、Hive 数仓建模与分布式计算（5 题）

## 1

请设计一套**支持高并发指标服务的数仓模型**，要求：

支持 OLAP 查询、支持自助分析、支持 API 输出、支持实时补数。

从分层、存储、引擎、服务化四个方面说明。

### **参考答案**

1. **分层**
    
    ODS → DWD（明细）→ DWS（主题宽表）→ ADS（指标服务）
2. **存储**

3. **引擎**
    
    Hive 离线计算 + Clickhouse/Doris 加速查询
4. **服务化**
    
    指标统一注册，DWS 层一数一源
    
    实时数据通过 Flink 写入 OLAP 引擎补全

## 2

手写 **用户 30 日留存 + 复购率 + 客单价** 合并指标 SQL，要求可直接上线。

### **参考答案**

```sql
WITH user_first_dt AS (
    SELECT user_id, MIN(dt) AS first_dt
    FROM dwd_order_info_d
    WHERE dt >= '2025-01-01'
    GROUP BY user_id
)
INSERT OVERWRITE TABLE ads_user_overview PARTITION (dt = '${dt}')
SELECT
    t1.first_dt,
    COUNT(DISTINCT t1.user_id) AS new_user_cnt,
    COUNT(DISTINCT t2.user_id) AS retain_30d_cnt,
    ROUND(COUNT(DISTINCT t2.user_id)/COUNT(DISTINCT t1.user_id),4) AS retention_30d,
    SUM(t2.order_amt) AS total_amt,
    COUNT(DISTINCT t2.order_id) AS total_order_cnt,
    ROUND(SUM(t2.order_amt)/COUNT(DISTINCT t2.user_id),2) AS avg_price
FROM user_first_dt t1
LEFT JOIN dwd_order_info_d t2
    ON t1.user_id = t2.user_id
    AND t2.dt = DATE_ADD(t1.first_dt, 30)
GROUP BY t1.first_dt;
```

## 3

一个任务明明数据量不大，但每次跑都**卡死、OOM、shuffle 失败**。

请写出你从**日志 → 数据分布 → 参数 → SQL 逻辑**的完整排查步骤。

### **参考答案**

1. 看 Yarn 日志：GC 频繁、shuffle 落盘失败、数据倾斜
2. 查看分组 key 分布：是否存在极端热点 key
3. 检查小文件是否爆炸，导致启动过载
4. 查看 SQL：是否存在笛卡尔积、暴力 distinct
5. 参数：内存、并行度、shuffle 分区数是否合理
6. 引擎：是否用了 mapjoin 但小表过大

## 4

什么是**数据集市**？和数据中台关系是什么？

如何避免 “数仓爆炸、指标重复、口径打架”？

### **参考答案**

- 数据集市：面向业务域的专用数据层（运营、销售、供应链）
- 中台统一标准与公共层，集市做定制化应用
    
    避免混乱：
- 一数一源，指标统一注册
- DWS 公共聚合，禁止 ADS 重复计算
- 血缘监控，自动识别重复指标
- 建立指标 owner 制度

# 三、向量数据库 Milvus & AI 数据中台（4 题）

## 1

Milvus 集群上线后出现**周期性查询毛刺、RT 突增、插入卡顿**。

请给出排查与优化方案。

### **参考答案**

1. 索引构建阻塞查询，关闭实时构建改为后台构建
2. segment 合并导致 IO 抖动，调整合并策略
3. 内存不足导致 swap，扩容或降低 HNSW 内存占用
4. 插入与查询争抢资源，分写节点与查节点
5. 连接池耗尽，调整客户端参数

---

## 2

向量库与数仓数据**经常对不上**：数仓有用户，Milvus 没有；或者向量过期。

设计一套**自动化对齐 + 对账 + 修复**机制。

### **参考答案**

1. 建立 `user_id` 唯一映射表
2. 每日数仓全量用户与 Milvus 主键对比
3. 缺失用户自动触发补推任务
4. 按更新时间标记过期向量，自动删除 / 更新
5. 周级全量重建，保证最终一致

## 3

在数据中台里，**向量数据属于结构化还是非结构化数据？应该怎么治理？**

包括：权限、脱敏、生命周期、质量监控。

### **参考答案**

- 向量属于非结构化特征数据
    
    治理：

1. 权限：与业务 ID 权限绑定
2. 脱敏：向量不直接泄露隐私，但需监控原始文本
3. 生命周期：冷向量自动下线、降精度、归档
4. 质量：监控向量维度一致性、空向量、召回率

# 四、综合架构 & 治理压力题（1 题）

## 面试官：

“如果让你从零到一搭建一套**可规模化、可治理、可复用**的数据中台，你前三个月怎么规划？每个阶段产出什么？”

### **参考答案**

1. **第 1 个月：现状调研 + 标准制定**
    
    - 梳理业务系统、数据源、痛点
    - 制定数据标准、命名规范、建模规范
    - 搭建元数据采集与数据地图
    
2. **第 2 个月：数仓分层建设 + 核心指标统一**
    
    - 搭建 ODS-DWD-DWS-ADS 标准分层
    - 建设核心主题：用户、商品、订单
    - 统一核心指标口径
    
3. **第 3 个月：治理体系 + 服务化上线**
    
    - 数据质量平台、监控告警、对账机制
    - 指标服务 API + 自助查询
    - 流程规范上线：接入、建模、变更、下线
