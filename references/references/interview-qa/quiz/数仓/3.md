# 一、MySQL 数据库设计与治理（5 题）

## 1. 请设计一张「用户行为日志表」（MySQL），用于数据中台同步到数仓

要求：

- 支持高并发写入
- 支持增量同步
- 支持数据治理（可追溯、可删除、可对账）
- 给出字段、主键、索引、引擎、分区思路

### 参考答案

```sql
CREATE TABLE user_behavior_log (
    id                  BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '雪花ID/自增ID',
    user_id             VARCHAR(64) NOT NULL COMMENT '用户ID',
    event_type          VARCHAR(32) NOT NULL COMMENT '事件类型:click/view/pay/cart',
    page_id             VARCHAR(64) NULL COMMENT '页面ID',
    item_id             VARCHAR(64) NULL COMMENT '商品ID',
    behavior_time       DATETIME NOT NULL COMMENT '行为时间',
    client_type         TINYINT NOT NULL COMMENT '1-app 2-h5 3-pc 4-mini',
    device_id           VARCHAR(128) NULL COMMENT '设备ID',
    source_channel      VARCHAR(64) NULL COMMENT '渠道',
    is_deleted          TINYINT NOT NULL DEFAULT 0 COMMENT '0-正常 1-删除',
    create_time         DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time         DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    KEY idx_user_time (user_id, behavior_time),
    KEY idx_event_time (event_type, behavior_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT '用户行为日志表';
```

设计要点：

- 用 `behavior_time` 做业务时间，方便增量同步
- 通用审计字段：`is_deleted、create_time、update_time`
- 索引只建高频查询组合索引，不冗余
- 大数据量可按 `behavior_time` 按月分区（或水平分表）
- 同步数仓使用 `update_time/behavior_time` 做增量抽取

## 2. 手写 SQL：统计每个用户每日首次访问时间、访问次数

表：user_behavior_log

### 参考答案

```sql
SELECT
    user_id,
    DATE(behavior_time) AS dt,
    COUNT(*) AS pv_count,
    MIN(behavior_time) AS first_visit_time
FROM user_behavior_log
WHERE is_deleted = 0
GROUP BY user_id, DATE(behavior_time);
```

## 3. 什么是回表查询？如何通过索引优化避免？

### 参考答案

- 普通二级索引查到主键后，需要再回聚簇索引查整行数据 = 回表。
- 优化：
    
    1. 建立**覆盖索引**，把查询字段全部放进索引
    2. 避免 `select *`
    3. 合理使用联合索引
    

示例：

```sql
-- 覆盖索引，不需要回表
SELECT user_id, behavior_time FROM user_behavior_log 
WHERE event_type = 'click' AND behavior_time >= '2025-01-01';
```

## 4. MySQL 数据治理中，如何做数据脱敏？举例实现手机号、身份证脱敏

### 参考答案

常用方案：

1. **查询时脱敏**（SQL 层）
2. **存储加密**（AES）
3. **中台 ETL 脱敏写入数仓**

SQL 示例：

```sql
-- 手机号脱敏
SELECT CONCAT(SUBSTRING(phone,1,3),'****',SUBSTRING(phone,8,4)) AS phone_mask

-- 身份证脱敏
SELECT CONCAT(SUBSTRING(id_card,1,6),'********',SUBSTRING(id_card,15,4)) AS id_mask
```

治理规范：

- 敏感字段必须打标签
- 权限分级：普通开发不可查明文
- 数仓层统一脱敏落地

## 5. 数据中台从 MySQL 同步数据，出现重复同步、数据不一致怎么解决？

### 参考答案

- 增量依据必须唯一有序：`behavior_time / update_time / binlog`
- 增加**唯一业务键**，写入数仓支持幂等
- 记录同步位点（offset），断点续传
- 每日行数对账、指标对账
- 重复数据用 `row_number()` 去重

# 二、Hive 数仓建模与 SQL 实战（5 题）

## 1. 请写出 ODS → DWD 用户行为表的清洗 SQL

要求：

- 按 dt 分区
- 去重
- 空值过滤
- 字段标准化

### 参考答案

```sql
INSERT OVERWRITE TABLE dwd_user_behavior_d PARTITION (dt = '${dt}')
SELECT
    id,
    user_id,
    event_type,
    item_id,
    behavior_time,
    client_type,
    device_id
FROM (
    SELECT
        *,
        ROW_NUMBER() OVER (PARTITION BY id ORDER BY update_time DESC) AS rn
    FROM ods_user_behavior_log
    WHERE dt = '${dt}'
) t
WHERE rn = 1
  AND user_id IS NOT NULL
  AND behavior_time IS NOT NULL;
```

## 3. 什么是数据倾斜？写出一个真实倾斜场景 + 完整优化 SQL

例如：group by user_id 出现超级大用户

### 参考答案

**现象**：某几个 user_id 占比 50% 数据，个别 reducer 卡死。

**优化 SQL（加盐打散）**：

```sql
-- 第一步加盐局部聚合
SELECT
    user_id,
    SUM(cnt) AS total_pv
FROM (
    SELECT
        CONCAT(user_id, '_', FLOOR(RAND() * 10)) AS user_rand,
        user_id,
        COUNT(*) AS cnt
    FROM dwd_user_behavior_d
    WHERE dt = '${dt}'
    GROUP BY user_id, CONCAT(user_id, '_', FLOOR(RAND() * 10))
) t1
GROUP BY user_id;
```

## 4. 数仓拉链表（SCD2）实现：用户信息变化如何存储？手写 SQL

### 参考答案

```sql
-- 拉链表核心：start_date end_date is_current
INSERT OVERWRITE TABLE dim_user_zip
SELECT
    user_id,
    user_name,
    user_level,
    start_date,
    end_date,
    is_current
FROM (
    -- 历史数据
    SELECT 
        user_id,user_name,user_level,
        start_date,
        CASE WHEN end_date = '9999-12-31' THEN DATE_ADD('${dt}',-1) ELSE end_date END AS end_date,
        0 AS is_current
    FROM dim_user_zip
    WHERE is_current = 1
    
    UNION ALL
    
    -- 新增当日变化数据
    SELECT
        user_id,user_name,user_level,
        '${dt}' AS start_date,
        '9999-12-31' AS end_date,
        1 AS is_current
    FROM dwd_user_info_d
    WHERE dt = '${dt}'
) t;
```

# 三、向量数据库 Milvus 实战（4 题）

## 1. 向量数据库适合解决什么问题？为什么不直接用 MySQL/Hive？

### 参考答案

- 适合：高维向量相似度检索（Embedding、图片、语音、用户向量）
- 不适合结构化 SQL 查询
- MySQL/Hive 暴力匹配 O (n)，Milvus 用 ANN 索引毫秒级查询

---

## 2. Milvus 核心流程：建库 → 建集合 → 创建索引 → 插入 → 查询

请写出关键步骤（文字描述即可）

### 参考答案

1. 创建 Collection，指定向量维度、主键字段
2. 建立向量字段（FLOAT_VECTOR）
3. 选择索引：HNSW/IVF_FLAT/IVF_SQ8
4. 批量插入向量 + 业务 ID
5. load collection 到内存
6. 输入向量执行 ANN 搜索，返回 topK 相似结果

---

## 3. 向量索引怎么选？请给中台场景选型

- 低延迟高 QPS
- 海量数据（亿级）
- 内存有限

### 参考答案

- 低延迟：**HNSW**
- 海量 + 内存有限：**IVF_SQ8**
- 中小规模 + 均衡：**IVF_FLAT**
- 极小数据量精准检索：**FLAT**

---

## 4. 数仓如何批量向 Milvus 写入向量？如何保证一致性？

### 参考答案

流程：

1. Hive/Spark 读取 DWD 文本 / 用户特征
2. 调用 Embedding 模型生成向量
3. 通过 Milvus SDK 批量写入
4. 建立业务 id <-> vector id 映射表

一致性治理：

- 按 dt 分区增量写入
- 重复写入覆盖（幂等）
- 每日向量数量与数仓行数对账

# 四、综合治理题（1 题）

## 请简述你对「数据中台数据治理体系」的理解，至少包含：

- 数据标准
- 数据质量
- 元数据 & 血缘
- 指标口径统一
- 数据安全与脱敏

### 参考答案

1. **数据标准**
    
    统一表名字段、枚举字典、数据类型、命名规范。
    
2. **数据质量**
    
    监控空值、重复、值域异常、行数波动、延迟告警，用规则引擎自动校验。
    
3. **元数据 & 血缘**
    
    全链路血缘追踪，表 / 字段 / 指标依赖可视化，方便变更影响分析。
    
4. **指标口径统一**
    
    指标字典化，DWS 层公共聚合，ADS 层禁止重复计算，确保一数一源。
    
5. **数据安全**
    
    敏感数据分级、字段脱敏、行权限控制、操作审计、分级授权。
