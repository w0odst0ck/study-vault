# 数据中台 / 数据治理工程师 模拟面试套卷（第三套・5 题）

## 题目 1：数据质量监控体系建设（实战 + 治理）

**问题**：你在工厂数据中台项目中负责数据质量保障，请说明如何搭建一套完整的数据质量监控体系？包含哪些核心维度、监控工具、异常处理流程，以及如何落地到日常运维？

## **参考答案**

### 一、核心监控维度（4 大维度）

- **完整性**：关键字段非空校验、记录条数一致性、必选字段缺失率（如`order_id`、`product_id`）
- **准确性**：数值范围校验（如产量≥0、金额非负）、业务规则校验（如良率≤100%）、重复数据检测
- **一致性**：跨表 / 跨系统数据一致性（如订单数 = 发货数 + 未发货数）、指标口径统一
- **及时性**：数据延迟监控（如 ODS 层延迟 > 1 小时告警）、ETL 任务执行时长监控

### 二、工具与技术实现

- **DataWorks 内置质量规则**：配置表级 / 字段级校验规则，自动巡检、异常告警
- **SQL 脚本 + 调度**：编写 Hive/MySQL 校验 SQL，定时执行，结果写入质量监控表
- **Great Expectations**：定义数据期望（如`expect_column_values_to_be_between`），自动化校验
- **监控看板**：用 Streamlit 做质量大盘，展示每日异常率、修复率、趋势

### 三、异常处理流程（闭环）

1. **告警触发**：异常通过短信 / 邮件 / 钉钉通知负责人
2. **根因分析**：定位问题（源系统脏数据、ETL 脚本 Bug、任务失败）
3. **数据修复**：回滚重跑、补数、清洗脏数据、隔离问题数据
4. **根因解决**：优化规则、修复脚本、规范源系统、完善 SOP
5. **复盘优化**：每周复盘，新增规则、降低重复异常

### 四、日常落地

- 分级告警：致命（立即处理）、严重（2 小时）、一般（当日）
- 纳入 KPI：数据质量达标率、异常修复及时率
- 定期审计：每月全量校验，输出质量报告

## 题目 2：Hive/Spark 数据倾斜实战解决（高频难点）

**问题**：你在处理工厂千万级数据时遇到数据倾斜，请说明如何发现、分析、解决 Group By 与 Join 倾斜？结合项目给出具体方案与效果。

## **参考答案**

### 一、如何发现倾斜（判断依据）

- 现象：多数 Task 快、少数 Task 极慢（卡住）、任务超时失败
- 监控：Spark UI 查看 Shuffle Read/Write 倾斜、Hive 日志查看 Reduce 分布
- 统计：`SELECT key,COUNT(*) FROM table GROUP BY key ORDER BY 2 DESC LIMIT 10`查热点 Key

### 二、Group By 倾斜（如按产品 / 车间统计）

**原因**：少数热门产品 / 车间数据量远大于其他

**解决方案**

1. **参数调优**

    ```sql
    set hive.groupby.skewindata=true; -- 开启倾斜优化
    set hive.map.aggr=true; -- Map端预聚合
    ```
    
2. **加盐两阶段聚合（项目实战）**

    ```sql
    -- 1. 加随机盐（0-9）局部聚合
    SELECT CONCAT(salt,'_',product_id),SUM(amount)
    FROM (SELECT product_id, FLOOR(RAND()*10) AS salt, amount FROM table) t
    GROUP BY CONCAT(salt,'_',product_id);
    -- 2. 去盐全局聚合
    SELECT split(new_id,'_')[1],SUM(amount) FROM temp GROUP BY split(new_id,'_')[1];
    ```
    

**效果**：单 Task 从 2 小时→10 分钟，无超时

### 三、Join 倾斜（如订单表 Join 产品表）

**原因**：热门产品 Key 集中、空值 / 无效值过多

**解决方案**

1. **MapJoin（小表广播）**

    ```sql
    set hive.auto.convert.join=true;
    SELECT /*+ MAPJOIN(product) */ o.* FROM order o JOIN product p ON o.product_id=p.id;
    ```
    
2. **热点 Key 分离处理**
    
    - 提取热点 Key 单独 Join，非热点正常 Join
    - 空值 Key 赋随机值打散，避免集中
        
        **效果**：大表 Join 从 3 小时→40 分钟
    

### 四、项目成果

通过倾斜优化，ETL 整体耗时降低 60%，集群资源利用率提升 40%，保障看板实时性

## 题目 3：元数据管理与数据血缘应用（治理进阶）

**问题**：请说明元数据管理的核心价值、在你项目中的落地内容，以及数据血缘如何支撑数据治理、问题排查与合规？

## **参考答案**

### 一、元数据核心价值

- **数据可理解**：表 / 字段业务含义、数据来源、计算逻辑、负责人
- **数据可管理**：全生命周期追踪、变更影响分析、权限管控
- **数据可复用**：快速找表、减少重复开发、统一口径

### 二、项目落地内容（DataWorks + 自研）

1. **技术元数据**
    
    - 表结构、字段类型、分区、存储格式、数据量、更新频率
    - ETL 任务依赖、调度周期、运行日志
    
2. **业务元数据**
    
    - 业务主题（生产 / 库存 / 订单）、指标定义、口径、使用场景
    - 负责人、使用部门、安全分级
    
3. **管理功能**
    
    - 元数据查询、数据字典导出、变更通知、血缘图谱
    

### 三、数据血缘核心应用

1. **问题快速排查**
    
    - 看板数据异常→向上追溯源表 / ETL 脚本→定位故障点
    - 例：库存数异常→发现 ODS 同步失败→修复任务
    
2. **变更影响分析**
    
    - 字段修改→查看下游依赖表 / 任务 / 看板→评估影响、通知负责人
    - 避免盲目变更导致全链路故障
    
3. **合规与审计**
    
    - 敏感数据（客户 / 金额）全链路追踪，确保脱敏、权限合规
    - 满足内外部审计数据溯源要求
    
4. **成本优化**
    
    - 识别无访问冷表、下线无效任务、释放存储 / 计算资源
    

### 四、价值

数据问题排查从 4 小时→30 分钟，变更故障降低 80%，合规审计通过率 100%

## 题目 4：RAG 系统检索优化（Milvus+BM25）

**问题**：你在法律 AI 项目中用 Milvus+BM25 优化检索，请说明 RAG 检索不准的常见原因、四层优化方案，以及在项目中的具体落地与效果？

## **参考答案**

### 一、检索不准常见原因

- 文档切分不合理：固定长度割裂语义、无重叠丢失上下文
- Embedding 模型泛化：垂直领域（法律）语义捕捉差
- 检索单一：仅关键词 / 仅向量，覆盖不全
- 无排序优化：TopN 结果相关性差

### 二、四层优化方案（项目实战）

#### 1. 文档分块优化（Chunking）

- **策略**：语义分块（按法律条款 / 章节）+ 10% 重叠（50 tokens）
- **长度**：512–768 tokens（适配法律文本密度）
- **效果**：避免语义断裂，关键信息完整保留

#### 2. Embedding 模型优化

- **选型**：用 BGE-large-zh（中文语义强）替代通用模型
- **领域微调**：用法律法规 / 案例数据微调，提升专业术语相似度
- **效果**：语义匹配准确率提升 35%

#### 3. 混合检索架构（Milvus+BM25）

- **BM25**：关键词精准匹配（法条编号、专业术语）
- **Milvus 向量**：语义相似度（相似案例、相近条款）
- **融合**：加权融合（BM25:0.4 + 向量：0.6）→重排序→取 Top5
- **效果**：召回率从 65%→92%，准确率从 70%→89%

#### 4. 后处理与过滤

- **元数据过滤**：按发布日期、法规类型、效力状态筛选
- **相似度阈值**：低于 0.7 过滤，避免低相关干扰
- **去重**：相似内容合并，保留最相关

### 三、项目成果

- 法律 AI 问答准确率从 72%→91%
- 平均响应时间从 800ms→300ms
- 用户满意度提升，支撑金融合规场景落地

## 题目 5：跨部门数据治理协同（管理 + 业务）

**问题**：数据治理常遇跨部门协作难（业务不配合、口径冲突、权责不清），你在工厂项目中如何推动落地？给出组织、流程、工具、激励方案与成果。

## **参考答案**

### 一、核心痛点

- 业务认为治理增加工作量、无直接价值
- 口径不统一：生产 / 销售 / 库存对 “产量”“库存” 定义不同
- 权责不清：出问题互相推诿、无人负责
- 数据孤岛：部门不愿共享数据

### 二、推动方案（四维落地）

#### 1. 组织保障（成立治理小组）

- **治理委员会**：IT + 业务负责人（生产 / 销售 / 财务），决策优先级
- **数据 Owner**：各部门指定 1 人，负责本部门数据标准、质量、授权
- **数据管家**：数据团队执行落地、技术支持、培训

#### 2. 流程机制（标准化）

- **需求评审**：新指标 / 表必须经治理委员会评审，统一口径
- **变更审批**：表结构 / ETL 变更→申请→评估影响→审批→执行
- **问题闭环**：质量问题→派单 Owner→整改→验收→归档

#### 3. 价值驱动（让业务受益）

- **自助分析**：可视化中台，业务自主查数、做报表，不用等数据团队
- **效率提升**：统一口径后，跨部门报表核对时间从 2 天→2 小时
- **问题预警**：数据异常自动通知，及时发现生产 / 库存问题

#### 4. 激励与考核

- **KPI 绑定**：数据质量、共享率纳入部门 / 个人 KPI
- **评优机制**：季度优秀数据 Owner 表彰、奖励
- **负面约束**：不配合影响项目进度，纳入绩效扣分

#### 5. 工具赋能

- DataWorks 权限管控、元数据查询、质量告警、血缘分析
- 数据门户：统一入口查字典、提需求、看质量

### 三、项目成果

- 跨部门数据冲突减少 90%，口径统一率 100%
- 数据共享率从 40%→90%，打破孤岛
- 业务满意度从 50%→95%，治理全面落地
