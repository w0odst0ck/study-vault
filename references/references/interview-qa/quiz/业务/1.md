# 数据中台 / 数据治理工程师 模拟面试套卷（5 题）

## 题目 1：数仓建模与分层设计

**问题**：你在忠新防护的项目中，基于 Hive 搭建了生产 / 库存 / 订单主题的数仓分层（ODS/DWD/DWS），请你详细说明这三层的核心定位、设计原则，以及你在项目中针对工厂业务做的针对性优化？

### **参考答案**：

1. **分层定位与设计原则**
    
    - **ODS 层（原始数据层）**：核心定位是**原样落地源数据**，是数仓的数据源入口。设计原则：保持与源系统数据结构、粒度完全一致，仅做简单的清洗（去重、格式校验），不做业务逻辑加工；按时间分区存储，保留全量历史数据，用于数据回溯。
    - **DWD 层（明细数据层）**：核心定位是**业务明细数据的标准化**，对 ODS 数据做清洗、维度退化、空值处理、脏数据过滤，输出符合业务定义的原子粒度明细数据。设计原则：高内聚低耦合，按业务主题（生产、库存、订单）分表，保留最细粒度的业务过程数据，为上层分析提供统一的明细基础。
    - **DWS 层（汇总数据层）**：核心定位是**面向分析的主题汇总**，基于 DWD 明细数据，按业务维度（时间、车间、产品、客户等）做聚合计算，生成面向业务场景的汇总宽表。设计原则：面向分析需求建模，复用性优先，减少重复计算，支撑 BI 看板、业务报表等上层应用。
    
2. **工厂业务针对性优化**
    
    - 针对工厂多源异构数据源（ERP、MES、WMS 等），在 ODS 层做了**异构数据统一接入**，通过 DataWorks 同步工具实现不同系统数据的批量拉取，解决了数据来源分散的问题。
    - 在 DWD 层针对工厂生产数据的特点，做了**生产工序维度的标准化**，统一了不同车间的工序编码、物料编码，解决了业务口径不一致的问题。
    - 在 DWS 层针对库存、订单的高频分析场景，做了**预聚合优化**，按天 / 周 / 月提前生成库存周转、订单履约等汇总指标，将 BI 看板的查询效率提升了 40% 以上，同时减少了实时计算的资源消耗。
    - 制定了完整的数据规范 SOP，明确了各层表的命名规则、字段定义、更新频率，支撑了跨部门数据的统一调用。

## 题目 2：ETL 流程自动化与性能优化

**问题**：你在项目中用 Python（Pandas/Numpy）搭建了自动化 ETL 数据管道，处理 5 + 异构数据源，日均处理 1 万条数据，请你说明 ETL 的核心流程，以及你在项目中做了哪些性能优化？

### **参考答案**：

1. **ETL 核心流程**
    
    - **Extract（抽取）**：从 MySQL、Excel、CSV 等 5 + 异构数据源中，批量抽取生产、库存、订单等业务数据，支持全量抽取和增量抽取（基于时间戳 / 自增 ID）。
    - **Transform（转换）**：对抽取的数据进行清洗（去重、缺失值填充、异常值过滤）、格式转换、特征提取、维度关联，完成数据的标准化加工。
    - **Load（加载）**：将加工完成的数据加载到数仓对应分层（ODS/DWD/DWS），支持全量覆盖和增量追加，同时完成数据校验，确保数据准确性。
    - **自动化调度**：通过 DataWorks 的调度节点，配置定时任务，实现 ETL 流程的无人值守自动运行，同时添加了告警机制，任务失败时自动通知。
    
2. **性能优化措施**
    
    - **数据处理优化**：使用 Pandas 的向量化运算替代循环操作，用 Numpy 做高性能数值计算，将单批次数据处理效率提升了 30%；针对大表数据，采用分块读取（`chunksize`）的方式，避免内存溢出。
    - **增量同步优化**：针对日均 1 万条数据的场景，放弃全量同步，采用基于时间戳的增量抽取，仅同步当日新增 / 变更数据，减少了数据传输量和处理时间，将 ETL 耗时从 2 小时缩短到 20 分钟以内。
    - **异构数据源适配优化**：封装了统一的数据源读取接口，支持不同数据源的快速接入，新增数据源的接入时间从 1 天缩短到 2 小时，提升了可扩展性。
    - **任务调度优化**：在 DataWorks 中配置了任务依赖，按数据流向串行调度，避免了资源争抢；同时添加了数据校验节点，在加载前做数据完整性校验，避免脏数据入库。

## 题目 3：DataWorks 在数据中台项目中的应用

**问题**：你的简历中多次提到阿里云 DataWorks，包括实时数据同步、建模、ETL 开发，请你说明 DataWorks 在你项目中的核心作用，以及你用它完成的关键工作？

### **参考答案**：

DataWorks 是我搭建工厂全域数据中台的核心工具，核心作用是**一站式数据开发与治理**，覆盖了数据同步、建模、ETL、调度、运维全流程，具体关键工作如下：

1. **数据同步与集成**
    
    - 用 DataWorks 的数据同步功能，搭建了 MySQL 业务库到数仓的实时 / 离线同步链路，实现了生产、库存、订单等业务数据的实时采集、清洗、加载，支撑了业务数据的实时性需求。
    - 针对异构数据源，用 DataWorks 的 Python 节点，封装了自定义同步逻辑，实现了 Excel、CSV 等非结构化数据的批量导入，完成了千万级数据的标准化治理。
    
2. **数仓建模与分层建设**
    
    - 依托 DataWorks 的数据建模能力，完成了 ODS-DWD-DWS 三层数仓的表结构设计，落地了 10 + 核心表，制定了数据规范 SOP，支撑了跨部门数据的统一调用。
    - 结合 MaxCompute 计算引擎，完成了分布式数据的导入与治理，实现了千万级数据的高效存储与计算。
    
3. **ETL 自动化与调度**
    
    - 用 DataWorks 的任务调度功能，搭建了自动化 ETL 管道，配置了定时任务，实现了数据处理流程的无人值守运行，日均处理 1 万条数据，数据处理效率提升 20%。
    - 用 Python 节点结合 Pandas/Numpy，完成了复杂的数据清洗、特征提取逻辑，实现了多源数据的自动化加工。
    
4. **数据服务与可视化支撑**
    
    - 用 DataWorks 的数据服务功能，封装了标准化数据接口，为 Streamlit 搭建的可视化中台提供了稳定的数据源，支撑了数据的实时化、可交互展示，减少了数据团队 30% 的重复取数工作量。
    - 完成了数据质量监控节点的配置，对核心表做了完整性、一致性校验，保障了数据质量。

## 题目 4：向量数据库 Milvus 与 RAG 技术的应用

**问题**：你在法律 AI 问答系统项目中，用 Milvus 搭建了向量数据库，结合 BM25 算法优化了数据检索流程，请你说明这个项目的技术架构，以及 Milvus 在其中的核心作用？

### **参考答案**：

1. **项目技术架构**
    
    - **数据层**：整合监管法规、历史违案例等多源非结构化金融数据，完成数据清洗、结构化、语义分词、同义词词库等全流程数据治理。
    - **向量层**：基于 Milvus 搭建标准化向量数据库，将文本数据转换为向量存储，制定了向量数据的入库、更新、检索规范，落地了金融法规领域的专属数据资产库。
    - **检索层**：采用**BM25 算法 + 向量检索**的混合检索架构，BM25 负责关键词精准匹配，向量检索负责语义相似度匹配，结合两者优势，提升检索的准确性和全面性。
    - **应用层**：打造轻量化数据检索中台，支撑金融融合场景的高效数据查询与复用，为法律 AI 问答系统提供数据支撑。
    
2. **Milvus 的核心作用**
    
    - **高性能向量存储与检索**：Milvus 是专为向量检索设计的分布式数据库，支持亿级向量的毫秒级检索，完美适配了金融法规数据的语义检索需求，解决了传统数据库无法高效处理向量数据的问题。
    - **数据资产化管理**：通过 Milvus 实现了非结构化文本数据的向量化存储，将零散的法规、案例数据转化为可检索、可复用的向量数据资产，支撑了数据的统一管理与更新。
    - **检索流程优化**：结合 BM25 算法，Milvus 的向量检索弥补了传统关键词检索的语义缺失问题，实现了 “关键词 + 语义” 的双维度检索，大幅提升了法律 AI 问答系统的答案准确性，检索效率提升了 30% 以上。
    - **可扩展性支撑**：Milvus 支持分布式部署，为后续数据量的增长、多模态数据（如合同扫描件）的接入提供了扩展空间，保障了系统的长期可用性。

## 题目 5：项目经历与业务价值结合（行为 + 技术综合题）

**问题**：你在忠新防护的工厂数据中台项目中，搭建了交互式可视化分析中台，减少了数据团队 30% 的重复取数工作量，请你详细说明这个项目的背景、你解决的核心问题、技术实现，以及最终的业务价值？

### **参考答案**：

1. **项目背景**
    
    工厂原有数据分散在 ERP、MES、WMS 等多个系统中，数据口径不统一，业务部门（生产、库存、销售）每次取数都需要数据团队手动跑 SQL、做报表，重复工作量大，数据时效性差，无法支撑生产 / 库存 / 订单的实时化管理需求。
2. **核心解决的问题**
    
    - 解决了多源数据分散、口径不统一的问题，实现了数据的统一治理与标准化。
    - 解决了业务部门取数依赖数据团队的问题，实现了数据的自助式分析。
    - 解决了数据可视化程度低、无法实时监控的问题，支撑了业务的实时化决策。
    
3. **技术实现**
    
    - **数据层**：基于 MySQL 搭建统一业务数据库，用 Hive 完成 ODS-DWD-DWS 三层数仓建模，用 DataWorks 搭建自动化 ETL 管道，实现多源数据的自动化清洗、同步，日均处理 1 万条数据，数据处理效率提升 20%。
    - **可视化层**：用 Matplotlib/Seaborn/Streamlit，开发一站式数据可视化分析中台，包含生产进度、库存周转、订单履约等 8 + 核心功能模块，实现了生产 / 库存 / 订单数据的实时化、可视化、可交互展示。
    - **服务层**：基于 FastAPI 封装标准化数据接口，实现了可视化中台与数仓的安全交互，保障了数据访问的安全性。
    
4. **业务价值**
    
    - **效率提升**：减少了数据团队 30% 的重复取数工作量，释放了数据团队的人力，让数据团队聚焦于更有价值的数据分析工作。
    - **业务赋能**：业务部门可以自助式查看数据、分析业务，无需等待数据团队，数据获取时效从天级提升到实时级，支撑了生产排程、库存优化、订单履约等业务决策。
    - **数据资产化**：完成了工厂全域数据的标准化治理，搭建了统一的数据中台，为后续的数字化转型（如生产预测、智能排产）奠定了数据基础。
    - **成本节约**：通过数据可视化，及时发现了库存积压、生产瓶颈等问题，帮助工厂优化了库存结构，降低了运营成本。
