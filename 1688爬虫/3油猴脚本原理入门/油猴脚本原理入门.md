## Part 1：油猴脚本（Tampermonkey）的本质（30 分钟）

### 1.1 它到底是什么黑科技？
- **官方定义**：用户脚本管理器。
- **爬虫视角的粗暴理解**：它是一个 **“浏览器内置的代码注入器”**。当你访问 `1688.com` 时，Tampermonkey 检测到 URL 匹配 `@match` 规则，就会把 `01_fetch.user.js` 里的 JavaScript 代码 **“偷偷塞进”** 当前网页的运行时环境中。

**关键区别**：
- 你在 F12 控制台（Console）里输入的代码，关掉页面就没了。
- 油猴脚本会**持久存在**，每次刷新页面自动运行，实现“无人值守翻页”。

### 1.2 为什么油猴脚本能绕过跨域（CORS）？
这是新手最困惑的一点。油猴脚本发出的 `GM_download` 或 `fetch` 请求，**并不是从网页的 JS 发起的，而是从浏览器扩展（Extension）的 Background 上下文发起的**。扩展拥有比普通网页更高的权限，可以无视 1688 的 `Access-Control-Allow-Origin` 限制，直接把整个页面的 HTML 源码下载到你的电脑硬盘。

> **工程师笔记**：这就像你让保安（油猴）去隔壁仓库拿东西，而不是自己（网页 JS）翻墙过去——保安有通行证。

## Part 2：解剖 `window.scrollTo` 与滚动加载（45 分钟）

### 2.1 打开你的 `01_fetch.user.js`，找到核心骨架
我们不需要读懂每一行 JS 语法，我带你只盯**三个关键词**：

```javascript
// 关键词 1：滚动控制
window.scrollTo(0, document.body.scrollHeight); 

// 关键词 2：定时器（节拍器）
setInterval(function() { ... }, 3000); 

// 关键词 3：下载
GM_download(url, name);
```

### 2.2 拆解 `window.scrollTo(0, document.body.scrollHeight)`
- **`window`**：代表当前的浏览器标签页窗口。
- **`scrollTo`**：一个内置函数，用来移动浏览器的**视口（Viewport）**。
- **`(0, document.body.scrollHeight)`**：
  - `0`：水平滚动条拉到最左边。
  - `document.body.scrollHeight`：获取整个网页内容的总高度（像素），垂直滚动条直接**拉到底部**。

**底层原理推演**：
当浏览器滚动条触底时，1688 页面上绑定的 `scroll` 事件监听器会被触发。它检测到 `window.scrollY + window.innerHeight >= document.body.scrollHeight - 200`（距离底部不足 200px），于是向服务器发起 Ajax 请求：“我要第 2 页的数据！”

**你的油猴脚本就是利用这个机制**：每隔 3 秒，强制把滚动条拽到底部，骗 1688 以为你在疯狂向下翻，从而源源不断地吐出新的商品卡片。

## Part 3：速度与激情 —— 0.5 秒 vs 3 秒的“反爬实战”（45 分钟）

这是今晚的高潮部分，我们将亲自扮演“暴躁爬虫工程师”，把 1688 惹恼。

### 3.1 安全准备（先备份！）
1. 复制一份 `01_fetch.user.js`，重命名为 `01_fetch.user_backup.js`（防止改崩了找不回原版）。
2. 确保你已经登录 1688 网页版（保持登录态，否则触发验证后更难处理）。

### 3.2 修改翻页间隔（找到 `setInterval`）
打开 `01_fetch.user.js`，找到类似这样的代码结构：

```javascript
// 原版（安全模式，3秒一翻）
var intervalTime = 3000; 
setInterval(function() {
    // ... 滚动 + 下载逻辑 ...
}, intervalTime);
```

**你的任务**：把 `3000`（毫秒）改成 `500`（毫秒，即 0.5 秒）。

```javascript
// 狂暴模式（0.5秒一翻）
var intervalTime = 500; 
```

### 3.3 运行“狂暴模式”并观察灾难现场
1. 在 Tampermonkey 管理面板中，确保你的修改版脚本已启用。
2. 打开 1688，搜索任意关键词（比如“投光灯”）。
3. 刷新页面，脚本开始自动运行。

**你会看到什么？**
- **前 2 页**：滚动飞快，商品如瀑布般涌入，下载正常。
- **第 3 ~ 4 页**：页面突然**不再加载新商品**，滚动条卡在中间。
- **紧接着**：屏幕中央弹出一个**滑块验证码**（“请按住滑块，拖动到最右边”），或者直接跳转到登录页要求重新扫码。

### 3.4 底层原理分析（为什么会被封？）
1688 的反爬系统（通常基于阿里云 WAF）在后台做了以下检测：

| 人类正常行为（3秒/次） | 机器人行为（0.5秒/次） |
| :--- | :--- |
| 滚动触底后，眼睛扫描商品，停留 2~3 秒。 | 滚动条像“抽搐”一样瞬间反复触底。 |
| 每次触底到加载完成，间隔不均匀（受网络影响）。 | 间隔毫秒级精准恒定（500ms 雷打不动）。 |
| **鼠标轨迹**：滚动有加速度，有停顿。 | **鼠标轨迹**：无鼠标移动，纯 JS 代码控制的跳跃。 |

服务器端根据这些特征，判定当前页面存在**非人类操作（Bot Behavior）**，于是将你的 Session 标记为“高风险”，直接中断数据返回并弹出验证码。

> **核心认知升华**：反爬的本质不是“能不能抓”，而是“像不像人”。你的油猴脚本用 0.5 秒暴露了机器的“精确性”和“单调性”，这就是为什么 README 里反复强调“合理控制频率”。

## 📌 今晚核心结论（请记在你的技术笔记里）

> **油猴脚本不是万能钥匙**。它是“浏览器环境内的自动化模拟器”。  
> **`window.scrollTo`** 控制的是“可视区域（Viewport）”，而不是后端接口。  
> **频率（Frequency）** 是反爬系统最敏感的神经，**随机延迟（Randomized Delay）** 是比固定间隔高明 100 倍的策略。
