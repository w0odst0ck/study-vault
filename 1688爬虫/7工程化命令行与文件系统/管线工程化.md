## Part 1：`argparse` —— 命令行参数的“路由器”（60 分钟）

### 1.1 核心认知纠正：`run1.py` 到底在做什么？
打开 `pipeline/run1.py`，你会发现它的顶部有这样的结构：

```python
import argparse
from pipeline import parse_02, clean_03, analyze_04  # 直接导入函数

def main():
    parser = argparse.ArgumentParser(description='1688 搜索页数据管线')
    parser.add_argument('--cat', type=str, required=True, help='品类名称，如 投光灯')
    args = parser.parse_args()
    
    # 🔥 关键：直接把接收到的 '--cat' 参数，当变量传给后续函数
    print(f"📂 开始处理品类: {args.cat}")
    
    parse_02.run(cat=args.cat)   # 注意：这里是函数调用，不是启动新进程！
    clean_03.run(cat=args.cat)
    analyze_04.run(cat=args.cat)

if __name__ == '__main__':
    main()
```

**底层真相**：
- **`02_parse.py`、`03_clean.py`** 内部都定义了一个接收 `cat` 参数的 `run()` 或 `main()` 函数。
- `run1.py` 只是作为“总入口”，把你在命令行敲的 `--cat 投光灯` 解析成 Python 变量，然后**在当前进程（Process）里依次调用这些函数**。
- **误区澄清**：这不是“A 脚本调用 B 脚本”，而是“主函数调用子函数”。所有代码运行在同一个 Python 解释器里，变量共享（前提是模块导入正确）。

### 1.2 手把手拆解 `argparse` 的“三部曲”
无论哪个脚本，写参数解析永远只有这三行套路：

```python
import argparse

# 第 1 步：创建解析器对象（给工具起个名）
parser = argparse.ArgumentParser(description='我的爬虫工具')

# 第 2 步：添加你想接收的参数
#   必填参数：required=True
#   默认参数：default='户外灯具'
#   类型转换：type=int  （默认接收的是字符串）
parser.add_argument('--cat', type=str, required=True, help='品类')
parser.add_argument('--page-limit', type=int, default=10, help='最大翻页数')
parser.add_argument('--debug', action='store_true', help='调试模式')  # 不加参数，只当开关

# 第 3 步：解析命令行的实际输入（sys.argv 的封装）
args = parser.parse_args()

# 使用结果
print(f"品类: {args.cat}")
print(f"页数限制: {args.page_limit}")
if args.debug:
    print("🐛 进入了调试模式")
```

**执行效果**：
```powershell
python run1.py --cat 筒灯 --page-limit 5 --debug
# 输出：品类: 筒灯，页数限制: 5，进入调试模式
```

### 1.3 如果没有传参，报错长什么样？（体验感拉满）
如果你只输入 `python run1.py` 而不带 `--cat`，由于 `required=True`，`argparse` 会立刻**终止程序**并打印帮助信息：
```
usage: run1.py [-h] --cat CAT
error: the following arguments are required: --cat
```
这比手写 `if len(sys.argv) < 2` 优雅一万倍，还能自动生成 `-h` 帮助菜单。

## Part 2：`pathlib.Path` —— 面向对象的文件系统（60 分钟）

### 2.1 告别 `os.path.join` 的“字符串噩梦”
老式写法：
```python
import os
folder = os.path.join('data', '投光灯', 'products_detail')
file_path = os.path.join(folder, 'all_products.csv')
```
**痛点**：`os.path` 返回的是字符串，拼接、判断存在、创建目录需要调用不同函数，极容易拼错斜杠（Windows 是 `\`，Linux 是 `/`）。

**新式写法（`pathlib`）**：
```python
from pathlib import Path

base_dir = Path('data') / '投光灯' / 'products_detail'  # 用 / 运算符拼接！
file_path = base_dir / 'all_products.csv'
```
**底层魔法**：`Path` 对象重载了 `__truediv__`（除法运算符），让路径拼接像数学公式一样直观。它内部会自动根据操作系统转换斜杠。

### 2.2 你项目的“杀手锏”：`mkdir(parents=True, exist_ok=True)`
这是全项目最关键的防御性代码，出现在所有 `run` 脚本的开头。

```python
from pathlib import Path

cat = '投光灯'
cat_dir = Path('data') / cat  # 假设不存在这个文件夹

# 🔥 这一行代码做了三件事：
# 1. parents=True：如果 'data' 文件夹都不存在，递归创建它！
# 2. exist_ok=True：如果文件夹已经存在，不报错，直接忽略。
# 3. 自动处理权限：如果路径里有非法字符，提前抛出异常。
cat_dir.mkdir(parents=True, exist_ok=True)
```

**实战实验（必做）**：
打开你的 Jupyter Notebook，手动操作：

```python
from pathlib import Path

# 实验 1：正常创建
p1 = Path('./test_temp/a/b/c')
p1.mkdir(parents=True, exist_ok=True)
print("✅ 创建成功")

# 实验 2：去掉 exist_ok，重复运行
p1.mkdir(parents=True)  
# 第二次运行会报错：FileExistsError: [Errno 17] File exists: 'test_temp/a/b/c'
# 加上 exist_ok=True 就永远不会报错。
```

**工程启示**：假设你要跑 10 个品类（投光灯、筒灯、射灯...），脚本运行时如果某个品类文件夹已存在，`exist_ok=True` 可以让程序**从容跳过**，而不是直接崩溃导致整条产线中断。

### 2.3 手动 `print` 出路径全家福（调试必备）
在 `run1.py` 里插入一段调试代码，看看 `Path` 对象到底藏了多少信息：

```python
cat = '投光灯'
base = Path('data') / cat

print(f"📍 绝对路径: {base.absolute()}")          # 完整磁盘路径
print(f"📁 父目录: {base.parent}")              # data
print(f"🧩 路径部分: {base.parts}")             # ('data', '投光灯')
print(f"🔍 是否存在: {base.exists()}")           # True/False

# 遍历目录下的所有 HTML 文件
for html_file in base.glob('*.html'):
    print(f"📄 找到文件: {html_file.name}")      # 只打印文件名，不带路径
```

**`glob` 的威力**：`base.glob('*.html')` 返回一个生成器，可以递归查找所有匹配的文件，比 `os.listdir` + `endswith` 简洁太多了。

## Part 3：`argparse` 与 `pathlib` 的“合体技”（45 分钟）

### 3.1 真正的工程写法（在 `run1.py` 里追踪数据流）
我们把两个知识点连起来看：

```python
import argparse
from pathlib import Path

def main():
    parser = argparse.ArgumentParser()
    parser.add_argument('--cat', required=True)
    args = parser.parse_args()

    # 1. 用 argparse 拿到品类名（字符串）
    category = args.cat
    
    # 2. 用 pathlib 构建动态路径
    data_dir = Path('data') / category
    input_html_dir = data_dir / 'raw_html'   # 假设油猴下载的 HTML 放这里
    
    # 3. 防御性创建目录
    input_html_dir.mkdir(parents=True, exist_ok=True)
    
    # 4. 检查是否有 HTML 文件，没有就友好报错
    html_files = list(input_html_dir.glob('*.html'))
    if not html_files:
        print(f"❌ 在 {input_html_dir} 下没有找到任何 HTML 文件！")
        return
    
    print(f"✅ 找到 {len(html_files)} 个 HTML 文件，开始解析...")
    # 接下来调用 parse_02.run(category)
```

**数据流可视化**：
```
命令行输入：--cat 投光灯
      ↓
argparse 解析为变量：category = "投光灯"
      ↓
pathlib 动态拼接：Path('data') / "投光灯" / "raw_html"
      ↓
脚本自动处理该路径下的所有文件
```

### 3.2 为什么你的项目需要这种“解耦”设计？
假设 1688 改版，你想把数据存到 `output/2026/投光灯/` 目录下。如果没有 `pathlib`，你需要改十几个文件里的字符串拼接。但现在，你只需要在 `config.py`（或每个脚本的开头）改一行：
```python
BASE_DATA_DIR = Path('output') / '2026'
```
所有依赖它的脚本自动适配，这就是**工程化维护**的威力。

## 🚨 今日翻车预警 & 急救包

| 现象 | 排查方法 |
| :--- | :--- |
| `ModuleNotFoundError: No module named 'pipeline'` | 你在项目根目录外运行了脚本。解决：`cd` 进入项目根目录（即 `1688-scrapy/`），再运行 `python -m pipeline.run1 --cat 投光灯`（用 `-m` 模块方式运行）。 |
| `TypeError: run() missing 1 required positional argument: 'cat'` | `run1.py` 里调用了 `parse_02.run()`，但 `02_parse.py` 里的 `run` 函数没写参数。检查 `02_parse.py` 的 `def run(cat):` 是否定义了入参。 |
| `Path('data/投光灯').mkdir` 报错 `OSError: [WinError 123]` | Windows 路径里有非法字符（比如表情符号或特殊空格）。检查你输入的 `--cat` 是否带了看不见的换行符。 |
