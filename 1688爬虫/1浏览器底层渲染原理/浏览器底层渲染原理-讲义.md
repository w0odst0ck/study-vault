## Part 1：核心概念（30 分钟）

### 1.1 浏览器拿到代码后的“流水线”
当你输入 `1688.com` 并回车，浏览器内部经历了以下**关键步骤**（爬虫只需关注前 3 步）：

| 步骤 | 名称 | 爬虫视角的解读 |
| :--- | :--- | :--- |
| ① | **网络请求** | 服务器返回一串**纯文本字符串**（即 HTML 源码）。 |
| ② | **字节 → 字符** | 浏览器根据 `Content-Encoding` 和 `<meta charset>` 决定用什么编码（**1688 是 GBK**）。 |
| ③ | **Tokens → DOM 树** | 浏览器将字符串解析成 **DOM（文档对象模型）**——这是一棵倒挂的树，`<html>` 是根，`<div>` 是子节点。 |
| ④ | 样式计算 & 布局 | 计算 CSS 决定元素位置（爬虫不需要）。 |
| ⑤ | **JavaScript 执行** | **爬虫最关注！** JS 可以随时增删改 DOM 树的节点。 |

> **爬虫核心定律**：油猴脚本和 Selenium 获取的，都是 **“步骤 ⑤ 执行完毕后”** 的最终 DOM 树。而 `requests.get()` 只能拿到步骤 ① 的原始字符串。

---

### 1.2 DOM 树 vs 源代码（90% 新手混淆点）
- **查看源代码（Ctrl+U）**：服务器直接返回的原始字节流，**不包含** JS 后来添加的内容。
- **Elements 面板（F12）**：浏览器**内存中实时维护**的 DOM 树，包含所有 JS 动态生成的内容。

**实战记忆法**：
在 1688 搜索页，如果“年销量”是滚动后才加载出来的，你按 `Ctrl+U` 是搜不到这个数字的，但在 `Elements` 里一定能搜到——这就是你的油猴脚本必须**先滚动**再下载 HTML 的根本原因。

---

### 1.3 1688 的“滚动加载”底层原理（Lazy Loading）
1688 搜索页并不是一次性返回 34 页数据，而是利用了 **Intersection Observer（交叉观察器）** 或简单的 `scroll` 事件监听。

**底层链路推演**：
1. 你滚动鼠标 → 触发浏览器 `scroll` 事件。
2. 绑定的 JS 函数判断：“滚动条距离底部是否小于 500px？”
3. 如果是，JS 发起一个新的 **XHR（Ajax）请求**（比如 `GET /search?page=2`）。
4. 服务器返回 **JSON 数据** 或 **HTML 片段**。
5. JS 把返回的数据拼接成 `<div class="offer-item">...</div>`，并用 `appendChild` 挂到 DOM 树上。

**这就是为什么你的油猴脚本必须有 `window.scrollTo` + `setInterval`**：你在人工模拟触发第 1、2 步。

### 2.3 模拟油猴的“等待”逻辑（理解 `setInterval`）
在 Console 面板输入以下代码，模拟油猴的滚动行为：

```javascript
// 这是一段你在油猴脚本里常见的 JS，今天我们只读不写
let page = 1;
const timer = setInterval(() => {
    window.scrollTo(0, document.body.scrollHeight); // 滚到底
    page++;
    if (page > 34) clearInterval(timer);
}, 3000);
```

- **观察 Network 面板**：每 3 秒，就会冒出一个新的 `page=2`、`page=3` 的请求。
- **爬虫原理升华**：如果你的 `setInterval` 间隔太短（比如 500ms），1688 的服务器会检测到异常高频，直接返回“无数据”或弹出滑块验证——这就是**反爬的频率控制**。
